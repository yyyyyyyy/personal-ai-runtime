# 经验文档：多文件任务执行规范

**用途：** 适用于任何项目、任何技术栈。记录可复用的工作方式与踩坑模式，不绑定具体仓库、模块名或提交号。

**与检查清单的关系：** 本文回答「为什么上次没一次做对」；可执行的步骤见 `docs/CONTRIBUTING.md`。

---

## 编码节奏

**核心规则：改 1–2 个文件 → 立刻跑静态检查与编译 → 再继续。**

- 每完成一个小阶段就验证，不要攒到最后。
- 类型/签名错误具有传染性：一处变更可能连锁到大量下游。
- 数秒的即时校验，往往比事后半小时排查便宜。

**增量编辑失控时，重写比修补便宜。** 同一文件连续 ≥3 次小块替换仍对不齐时，用版本控制取回基线后整文件重写，避免中间状态累积成重复定义或半截逻辑。

---

## 调试方法论：先看数据，再写代码

**核心原则：改变代码之前，先通过日志/数据库/API 确认「现在到底发生了什么」。**

### 为什么改了很多次还不对

每次修复失败的典型模式是：

```
看到症状 → 猜测原因 → 改代码 → 验证（通常只验语法） → 以为修好了 → 实际没修好
```

这个模式的根本缺陷在于**跳过了「确认现状」这一步**。没有元数据支撑的「猜测原因」≈ 抛硬币。

### 五步诊断法

按顺序执行，不要跳步：

| 步骤 | 做什么 | 工具（本项目示例） |
|------|--------|-------------------|
| **1. 确认症状** | 精确描述用户看到了什么，在哪一层看到 | 浏览器截图、终端日志 |
| **2. 查数据库** | 数据实际被写成了什么？状态机走到了哪里？ | `sqlite3 data/personal_ai.db "SELECT * FROM messages WHERE …"` |
| **3. 查日志** | 后端每一跳的输出是什么？异常发生在哪一步？ | `tail -f` uvicorn 日志、LLM call 记录表 |
| **4. 画数据流** | 从入口到出口，画出每一层的数据变换 | 白板或用注释标注：LLM → strip → save → SSE → fetch → render |
| **5. 定位断层** | 数据在哪一步从「预期」变成了「异常」？ | 对比期望输出与实际输出，找到偏差点 |

**关键信条：** 步骤 2-4 完成之前，不要写任何修复代码。

### 典型案例（审批后 LLM 不继续任务）

**错误做法（改了 3 次）：**
1. 看到标记泄漏 → 改前端 strip 函数（没用）
2. 标记消了但任务没继续 → 改前端渲染层（没用）
3. 查数据库才发现 `continue_after_tool_result` 产出空消息

**正确做法（一次到位）：**
1. `SELECT * FROM messages` → 看到 Message 9 存了纯标记文本
2. 对比前后消息 → 发现 Message 8 是 `tool` 结果，Message 9 是 `assistant` 但内容是标记
3. 定位到 `continue_after_tool_result` 的调用链
4. 手工 `strip_tool_markup(Message 9 内容)` → 结果 `""`
5. **结论：根因是 LLM 输出被剥离为空，而非剥离本身不完整**

### 区分症状与根因

| 症状 | 可能的根因（需逐一排除） |
|------|------------------------|
| 前端显示 `<｜tool_calls>` | A) 前端 strip 不完整 / B) 后端传了原始文本 / C) LLM 输出即标记 |
| 审批后对话停止 | A) LLM 空响应 / B) 消息未持久化 / C) 前端没渲染 / D) 网络断连 |
| 工具执行失败 | A) 参数错误 / B) 权限不足 / C) 工具未注册 / D) 超时 |

症状是「第一现场」，根因往往是**几层上游的某个调用**。每看到一个新症状，追问：「这个数据的上一跳是什么？它是不是已经坏了？」

### 按层次修复（自上而下）

修复时明确区分四个层次，不要混着改：

| 层次 | 例子 | 应该先修哪个？ |
|------|------|---------------|
| 源数据层 | LLM 输出了不该输出的内容 | **先修这个** |
| 传输层 | SSE/API 传递过程中数据被污染 | 源对了再查传输 |
| 存储层 | DB 写入了错误的值 | 源和传输对了再查存储 |
| 展示层 | 前端渲染了不该渲染的内容 | **最后修**（源正确时它自然不出现） |

**教训：** 本次 bug 花了两轮修展示层（前端 strip），但源头（`continue_after_tool_result` 的空响应）修好后，展示层的改动才真正生效。

### 改完代码后必须验证效果

不是跑通 `import` 就行。至少确认：

- [ ] 数据库写入了正确的值（查 SQLite）
- [ ] 日志没有新的异常（查 uvicorn 输出）
- [ ] 用户路径走通了（浏览器实际操作一次）
- [ ] 真实数据验证：用**出问题的同一条对话**重新触发，确认不再复现

## 变更影响分析

**改动公共接口之前，必须先搜影响范围。更改成本与调用方数量成正比。**

- 用全文搜索确认函数、类型、配置项的所有引用。
- 新增对外符号前，搜索是否重名、是否已有同类抽象。
- 改测试 fixture / 全局单例 / 共享配置后，跑**全量**测试，不只跑新增用例。

**模块级单例是隐形依赖网。** 改配置、数据库连接、认证状态时，先搜「谁 import 了谁」；`reload` 往往清不干净已缓存的引用，优先用可显式重置的工厂或测试专用 hook。

## 验证与审查：四层模型

不要停在「语法通过」就认为交付完成。

| 层级 | 检查什么 |
|------|----------|
| 语法与类型 | lint、formatter、类型检查、能否编译/构建 |
| 功能闭环 | 新代码是否被调用？用户路径是否走通？ |
| 数据流一致性 | 读写是否走设计规定的路径？安全叙事与实现是否一致？ |
| 运维一致性 | 配置模板、文档、示例、脚本、CI 是否与代码同步？ |

**测试全绿 ≠ 功能闭环。** 常见盲区：

- 新增了 API 或导出函数，但没有任何调用方。
- 服务端加了认证，客户端/桌面端/脚本未传凭证。
- UI 只测到「组件渲染」，未测到「用户操作后的副作用」。
- 集成测试 mock 了成功路径，却未覆盖真实 E2E 或专用验证脚本。

**交付前至少确认：**

- [ ] 静态检查与全量测试通过（范围与 CI 一致）
- [ ] 新增符号有引用；跨层改动走过端到端
- [ ] 配置默认值、环境变量示例、用户文档三者一致
- [ ] `git status` 无遗漏的未跟踪测试/配置文件（不能只看 `diff --stat`）

## 配置与环境变量

**同一变量在不同运行时里的加载路径可能不同。** Monorepo 或前后端分离时尤其常见：

| 运行时 | 常见陷阱 |
|--------|----------|
| 服务端配置框架 | 自动读 `.env`，与手动 `load_dotenv` 重复加载 |
| 前端构建工具 | 默认不读仓库根目录的 `.env` |
| 测试 runner | 需显式 `setenv` / fixture 注入，不会自动继承 shell |
| 容器 / Compose | 镜像内绑定地址与宿主机暴露方式不一致 |

动环境变量时，列出**所有**会读它的进程（Web、Worker、CLI、测试、桌面壳），逐一确认。

**安全类配置的默认策略建议 opt-in：** 认证、强制 HTTPS、破坏性操作二次确认等——默认关闭或默认最安全的主机绑定，并在启动日志里说明风险，避免 silently 扩大攻击面。

## 测试策略

**改共享测试基础设施 → 必须跑全量测试。** fixture、conftest、全局 mock 的小改动可能让大量用例在错误上下文里仍「通过」。

**集成测试与 E2E 要分层，不要一锅炖：**

| 测什么 | 适合方式 |
|--------|----------|
| 路由、参数校验、错误码 | HTTP/RPC 集成测试 + mock 下游 |
| 真实数据迁移、破坏性操作 | 独立验证脚本或隔离数据库的 E2E |
| 全局单例（DB、缓存、连接池） | 避免在共享集成测里真删库；成功路径可 mock |

**破坏性操作的真实执行会污染后续用例。**  wipe / restore / `DROP` 类 API 若在同事务、同进程、同单例里跑通，后面测试可能出现「表不存在」「连接已关闭」等幽灵失败。契约测 mock，重度 E2E 单独 job 或脚本。

**前端测试常见坑：**

- 严格模式 / 双次挂载 → `getAllBy*` 优于 `getBy*`。
- jsdom 缺浏览器 API → 在 `beforeAll` mock，勿在 `afterEach` `delete` prototype（易与 React 清理冲突）。
- 「弹窗出现」≠ 流程完成 → 需断言回调参数、请求体、后续 UI 状态。
- **React 19 StrictMode — 即使有 cleanup，同一文本可能出现在多个节点。** `getAllByText("对话")[0]` 可能比 `getByText("对话")` 更可靠，因为 "对话" 同时出现在导航标签和按钮文字中。
- **共享 setup 中的全局 `cleanup()` 是解决 DOM 累积污染的最优解。** 在每个测试文件的 `beforeEach` 中各自调用不如在 `setup.ts` 中注册一次 `afterEach(cleanup)`。

**扩大 CI 门禁前先量一下：** 把新包纳入 coverage 或 typecheck 范围时，先本地跑一遍，确认不会把总门禁打到失败；否则应分阶段纳入或先还债。

## 文档与一致性

**代码改了，周围的一切不会自动更新。** 至少搜索并核对：

- README / 快速开始
- 架构或威胁模型文档
- API 示例、curl、OpenAPI 描述
- 静态 mock 页、截图用的示例命令
- Compose / K8s / 部署模板中的端口与认证说明
- 各端客户端（Web、移动端、桌面壳、CLI）

**安全或认证改动后，用旧关键词全仓搜索**（如 query token、无确认码的导出、默认 `0.0.0.0`），清掉叙事漂移。

## 安全实现（通用原则）

### 按攻击面收口，不只按文件

修完一处漏洞后追问：**同一能力是否还有第二条路径？** 例如：HTTP 客户端已拦 SSRF，但 Shell 里仍可 `curl` 内网；文件读已限制目录，但写或列目录是否共用同一边界函数。

### 同类规则应集中实现

URL 校验、路径规范化、权限判断、审计日志——抽成单一模块供所有入口复用，避免复制粘贴后规则漂移。

### 命令执行

- 白名单 + `shell=True` 通常仍是假安全：管道、重定向、解释器 `-c` 可绕过首 token 检查。
- 优先：`shell=False` + 参数数组 + 禁止 shell 元字符 + 限制危险 flag。
- 跨平台注意：部分命令是 shell 内建而非独立可执行文件，测试时选跨平台命令。

### 路径与文件访问

- 不要用字符串 `startswith` 判断是否在允许目录内（前缀碰撞：`/allowed` vs `/allowed_evil`）。
- 用规范化后的路径 + 语言提供的「相对路径」API（如 `is_relative_to`、 `commonpath`）。
- **同一前缀族内要区分「密钥」与「模板」**（如 `.env` vs `.env.example`）：不要用一个宽泛的 `startswith` 规则覆盖整族文件名。

### 出站请求（SSRF）

- 限制 scheme；拒绝 loopback、私网、link-local、metadata 地址。
- 域名要解析后检查 IP，不只检查字面 hostname。
- 重定向目标再次校验。

### 敏感 API

- 导出、导入、删除等操作：操作级确认码或二次授权，不要只依赖「可选的全局认证」。
- 默认绑定 localhost；监听 `0.0.0.0` 时强制或强烈提示启用认证。

### 凭证传递

- 避免把 token 放在 URL query（日志、历史记录、Referer）。
- WebSocket 优先：子协议、首帧认证、或 `Authorization` 头（视栈而定）。
- 改认证方式时，列出**所有客户端**（Web、桌面、移动、脚本、第三方集成）。

### 多源策略名单

权限、风险等级、功能开关若在多处各写一份（代码、配置、文档、策略表），**必然漂移**。做法：

- 指定**唯一真源**（注册表、策略文件、代码生成）。
- 用契约测试断言各副本集合相等或包含关系。
- 别名、旧名不报错但会使防护静默失效——用测试锁死命名一致。

## 删除与剥离模块

删掉实验目录或废弃包之后，** Makefile、CI、验证脚本、文档不会自动更新**。剥离后应搜索旧模块名，并本地跑一遍原 CI 中的脚本目标，而不只跑单元测试。

## 技术陷阱速查（通用）

| 陷阱 | 表现 | 对策 |
|------|------|------|
| 配置单例 + `reload` | 部分模块仍持有旧配置 | 显式 reset 或依赖注入；列出所有 import 点 |
| 多次小块替换 | 重复定义、匹配失败 | 超 3 次改用整文件重写 |
| 测试只跑新增文件 | fixture 改动连锁失败未发现 | 改基础设施后全量测试 |
| `diff --stat` 当完成标准 | 新文件未 add，CI 漏测 | 看 `git status` |
| UI 只测渲染 | 回调/请求未接线也绿 | 断言副作用与参数 |
| 白名单 + shell 执行 | 首 token 合法即可绕过 | argv 执行 + 禁元字符 |
| 路径前缀字符串比较 | 相邻目录绕过 | 规范化 + 相对路径 API |
| 单点 SSRF 防护 | Shell/代理/另一 HTTP 客户端仍可达内网 | 共用出站 URL 策略 |
| 集成测真实 wipe | 污染全局 DB/缓存单例 | 契约 mock + 独立 E2E |
| 集成测真实破坏性导入 | 外键/约束导致失败或脏数据 | 分层测试 |
| HTTP 客户端 DELETE + body | 部分测试客户端不支持 `json=` | 用通用 `request` 方法 |
| 扩大 coverage 范围 | 总覆盖率跌破门禁 | 先本地量；分模块门禁或先补测 |
| 扩大 typecheck 范围 | 历史债务一次暴露 | 分阶段纳入或先还债 |
| PowerShell `&&` / heredoc | 命令解析失败 | 分步执行；`git commit` 多 `-m` |
| 只改主 README | 威胁模型/示例页仍写旧行为 | 安全改动后全仓搜旧关键词 |
| 容器内 `0.0.0.0` | 宿主机端口映射即暴露 | 文档 + 默认认证或绑定策略 |
| React 19 StrictMode 测试 DOM 累积 | 多测试文件间 DOM 不清洗，`getByText` 报 multiple elements | 在 `setup.ts` 中注册全局 `afterEach(cleanup)`；含重合文本时用 `getAllBy*[0]` |
| 类型包版本漂移 | `@types/pkg@^15` 对不上 runtime `pkg@^16` | 先确认 runtime 版本是否**实际内置**类型声明；若未内置，`@types/` 版本号不匹配也可用（类型接口通常向后兼容） |
| 模块拆分后残留导入 | 拆分出的方法不再调用某些常量/函数，但 import 未清理 | 拆分后搜索被提取的符号名，确认原文件不再有残留引用 |
| Monorepo 新增测试文件未 git add | 测试全绿但 CI 不跑新文件 | 每新增文件后立即 `git status`，确认被 tracked 或 stage |
| AI 编码工具在 Windows 上执行 shell | `&&`、heredoc 不兼容 PowerShell | 用 `;` 分隔命令；多行文本用 Write 工具写入文件而非 heredoc |
| **拆分模块后未跑 lint** | ruff 检查 import 排序失败，CI 挂 | 拆文件 → 立即 `ruff check` + `ruff --fix`，不要等提交后再修 |
| **测试 mock 数据字段漂移** | TS 接口新增必填字段，测试 mock 未同步，`tsc` 报 `TS2741` | mock 数据构造时对照接口定义逐字段检查；或使用 `satisfies` 约束 mock 类型 |
| **提交前未跑全量 CI** | 本地测试绿但远端 CI 红灯（lint/tsc 未跑） | 提交前跑与远端 CI **完全相同**的命令序列，包括 lint/typecheck/verify 脚本 |
| **Python mixin 触发 mypy attr-defined** | 把方法拆分到 Mixin 类后，mypy 报 17 个 `"Mixin" has no attribute "xxx"` — 因为 mypy 将 Mixin 视为独立类，不知道它会被混入 Kernel | 为 Mixin 声明一个 Protocol 接口，列出它依赖的宿主类属性（如 `emit_event`、`_db`），让 Mixin 继承该 Protocol。运行时 Protocol 零开销，仅向 mypy 声明契约 |
| **Makefile 的 ci-local 与远端 CI 不一致** | 本地跑 `make ci-local` 全绿，但 push 后远端 CI 挂 ruff/mypy — 因为 Makefile target 漏了 lint/typecheck 步骤 | 新增或修改 Makefile 后，**对照 `.github/workflows/ci.yml` 逐 job 校验**：确保 Makefile 中的命令序列是 CI 步骤的真子集；远端 CI 新增 step 时同步更新 Makefile |
| **新增写类 Capability 只改 hub** | `test_taint` 契约红：WRITE_CLASS_TOOLS ≠ policy.needs_user | 按「模式 7」清单同步 hub / policy / taint / 前端 / CI / 文档 |
| **patch 工具空 old_string** | `str.replace("", "X")` 在全文插入，文件损坏 | 显式拒绝空串；单测覆盖 |
| **`.env*` 前缀拦截** | `.env.example` 模板也被挡，配置文档流断 | 密钥文件与模板分开规则：挡 `.env` / `.env.*`，放行 `.env.example` |
| **安全默认名单被 env 覆盖** | 用户设保护路径 env 后 kernel 不再受保护 | 自定义项**追加**到默认名单，文档写清语义 |
| **写操作审批 UI 只有工具名** | 用户无法 review 实际改动 | ConfirmationDialog 展示 diff 或内容摘要，超长可展开 |
| **Brain prompt 当唯一安全层** | prompt 禁止改 kernel，但 write 仍能写 | Capability 层 protected paths 硬拦截 |
| **只改前端 UI 来「修复」后端数据问题** | 标记被隐藏，但 LLM 仍产出空响应，功能不闭环 | 先查数据库确认数据正确性，再决定修哪一层 |
| **未查数据库就开始猜根因** | 连修 N 轮都是猜测，浪费数小时 | 五步诊断法：症状 → 查库 → 查日志 → 画数据流 → 定位断层 |

## 任务模式（抽象场景）

以下不对应某个仓库，只描述反复出现的**任务形状**与**推荐节奏**。

### 模式 1：Review 驱动修复

1. 按严重程度列 findings  
2. 制定分阶段计划（Critical 优先）  
3. 每阶段：实现 → 窄测试 → 全量测试  
4. 自 Review：对照四层模型，不只信「测试绿了」  
5. 第二轮补：文档、次要客户端、第二条攻击路径  
6. 与 CI 对齐后提交  

**典型遗漏：** 只改了主路径客户端；配置示例与架构文档未同步；新测试文件未纳入版本控制。

### 模式 2：安全边界加固

1. 命令执行、出站 HTTP、文件路径、默认网络暴露、敏感 API  
2. 每类抽共享校验模块，所有入口复用  
3. 集成测覆盖拒绝路径；重度 E2E 用独立脚本  
4. 搜索是否还有其他工具/脚本可走同攻击面  

### 模式 3：认证 / 凭证传递变更

1. 列出所有传输层：HTTP 头、Cookie、WS 握手、CLI 参数、桌面壳  
2. HTTP + 长连接 + 各端客户端各测一条 happy path  
3. 全仓搜索旧传参方式（如 `?token=`）并清文档  

### 模式 4：配置与多运行时

1. 改默认值或 env 名 → 搜全仓引用  
2. 分别验证：服务启动、前端构建、测试、容器  
3. 示例 `.env` 与代码默认一致  

### 模式 5：提交前 CI 对齐

1. 本地跑与远端 CI **相同**的 lint、typecheck、test、coverage 门槛  
2. 若某步本地无法跑（需 `gh`、需 Linux），注明风险  
3. 提交前 `git status` 确认无遗漏未跟踪文件  
4. **提交前必须跑全量 CI 脚本，不只单元测试。** 本次提交漏跑了 `ruff` 和 `tsc`，导致 CI 红灯 3 处：
   - `ruff I001` — 拆模块后 import 排序乱了，`ruff --fix` 一键修复
   - `tsc TS2741` — mock 数据缺少接口新增字段
   - `tsc TS7016` — 误删了仍需的 `@types/` 包

   **编排建议**：在提交前用一个脚本或 Makefile target 串联全部检查，避免人工遗漏任一步。

5. **Makefile 的 ci-local 必须与 .github/workflows/ci.yml 同步。** 容易出现的漂移：远端 CI 新增了 lint/typecheck/schema-verify 等 step，但 Makefile target 未同步更新，导致本地 `make ci-local` 全绿但 push 后远端挂。每次远端 CI 变更后，locally diff 两个文件的命令列表并修复差异。

### 模式 6：Review 驱动的全栈渐进改进

适用场景：拿到一份 Review 报告，按优先级从 P1→P3 逐步落实改进。

1. **拆为清单，一次只做一层。** P1（短期）→ P2（中期）→ P3（长期），不跨层混做。
2. **每项改进完成后立即跑相关测试。** 不要攒齐所有改动再验证——拆模块后立刻跑边界检查 + 重建验证，新增测试文件后立刻跑 vitest。
3. **涉及依赖变更时先 `npm install` 确认包存在。** 盲目改 `package.json` 版本号可能选中不存在的版本。
4. **自 Review 改进本身：** 对照四层模型复查自己的改进是否引入了新问题。本次任务中，kernel 拆分后发现了 3 个残留的陈旧 import，在提交前清理。
5. **前端测试需要基础设施先行。** 本次在创建 3 个新测试文件之前，先修复了 `setup.ts` 中缺少的全局 `cleanup()`；否则每个新测试文件都会触发双重渲染污染。

**典型遗漏：** 改了 `package.json` 但未跑 `npm install`；拆模块后原文件遗留了已迁移的 import；新增测试文件未被 git 发现。

### 模式 7：Agent Coding / 写类 Capability 接入

适用场景：让 AI Agent 在受治理前提下读写代码或配置（非全自动「自进化」）。

1. **先划边界再开放能力。** 区分「可改的 App 代码」与「不可改的治理层（kernel、策略加载器、污点规则、真实密钥文件）」——在 Capability 层硬拦截，不只靠 system prompt。
2. **新增写类工具时的同步清单（多源策略必查）：**
   - 工具注册（hub / MCP）
   - 风险策略（`needs_user` / forbidden）
   - 污点写类集合（须与策略契约测试一致）
   - 敏感路由 / Critic / 自动编排跳过名单
   - 前端审批文案与 diff 预览
   - CI 内置工具集合与文档中的工具数量
3. **优先 `search-replace` / patch，整文件 write 作兜底。** 小粒度 patch 降低 LLM 重写整文件的出错率；审批 UI 应展示 `old → new` 或写入内容摘要。
4. **patch 实现的边界用例必测：**
   - 空 `old_string`（Python `"x".count("")` 会匹配全文）
   - 多处匹配且未设 `replace_all`
   - 前缀规则误伤模板文件（如 `.env.example` vs `.env.local`）
5. **环境变量覆盖默认安全名单时，用「追加」而非「替换」。** 否则用户加一条自定义保护路径时，可能意外丢掉默认的 kernel / 策略保护。
6. **import 时初始化的单例读配置 → 改 env 必须重启进程；** 测试里 patch `settings` 字段比 `reload` 更可靠。
7. **两轮 Review 节奏：** 第一轮抓安全/逻辑硬伤；第二轮抓文档漂移、审批 UX、集成测试与边界 case。不要指望一轮找全。

**典型遗漏：** 只注册 hub 未更新 `taint.py` 导致契约测试红；`.startswith(".env")` 挡住 `.env.example`；审批弹窗只显示路径不显示 diff；README 仍写旧工具数。

## 如何在本仓库落地

本文刻意不写具体路径。执行时：

- 可执行命令与项目门禁 → `docs/CONTRIBUTING.md` 中的「本项目映射」一节（若有）
- 架构契约、威胁模型 → 各项目自有文档
- 每次大任务结束后，只把**新的通用模式**追加到上文；项目轶事不写入本文

## 文档分工

| 文档 | 内容 | 适用范围 |
|------|------|----------|
| `POSTMORTEM.md`（本文） | 原则、模式、陷阱、抽象场景 | **任何项目** |
| `CONTRIBUTING.md` | 检查清单；可选「本项目映射」 | 全局清单 + 当前仓库落地 |

两者建议放在 `docs/` 并由 `.gitignore` 忽略时，作为**个人/团队的本地协作记忆**，复制到新项目时只需改「本项目映射」一节。
