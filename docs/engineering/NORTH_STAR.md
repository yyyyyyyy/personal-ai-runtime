## 1. Mission（使命）

构建一个**数据主权属于用户、行为可治理、可重建、可审计**的个人 AI 运行时（Personal AI Runtime）。

它不是「又一个 AI 助手应用」。它是一条**治理边界**——把不可信的 AI 代理（Agent）约束在
受管通道之内，让一切改变系统或外部世界的行为都留痕、可追溯、可撤销、可治理。

交付形态：一个**本地优先（local-first）、单用户、单进程**的运行时，用户的全部个人数据
（事件、状态、记忆）驻留在用户自己的机器上，可一键无损导出，永不被任何厂商锁定。

---

## 2. Long-term Vision（长期愿景）

运行时治理的不是计算资源，而是**用户数字生活中的事实、记忆、能力与行为**。

### 愿景的四个支柱

#### 支柱一：唯一真相（Single Source of Truth）

用户的人生事实以**不可变、有序、可重放的事件日志（Event Log）**为唯一真相。

- **State**（及同级结构化事实）是物化投影：丢掉投影，永远可以从事件流**确定性重建**（rebuild）。
- **Memory** 是从事实与对话**推导**而来的信念：丢掉投影，必须能够从事件流**重新推导**（re-derive）——
  语义上忠实于来源，**不要求**与历史上某次抽取结果字节级相同（抽取所用模型会变、会下线）。

丢掉事件流，则真相永久丢失。

用户应当能带着自己的事件日志走向任何一个兼容实现，**重建**结构化事实视图，并**再推导**出语义一致的记忆层。

#### 支柱二：不可越界（Governed Boundary）

任何 AI 代理——无论它今天叫什么、明天由谁实现——
**永远不能直接读写用户的存储、文件系统、网络**。它们只能通过受治理的受管通道访问世界。
这条边界是运行时的本质，不是「有状态」，而是「有边界」。

#### 支柱三：可治理的行为（Governable Action）

代理对外部世界的一切交互（改文件、跑命令、发邮件、联网……）都经过统一的**能力治理**，
受风险分级、用户审批、来源污点追踪的约束。
高风险操作必须有人类在环确认；被不可信内容污染的推理链不得自动触发破坏性动作。

#### 支柱四：可携带的人生（Portable Life）

用户与运行时相处的全部历史——对话、记忆、目标、决策、审批记录——是用户自己的资产，
不是平台的资产。导出、迁移、删除这些数据的权力始终属于用户，而非运行时实现。

## 3. Core Principles（核心原则）

这些原则**定义项目本身**。违反任何一条，就不再是这个项目。它们不随实现技术栈变化而改变。

### P1 · Event Log 是唯一事实来源

事件日志（Event Log）是系统的黑匣子：只追加、不可变、全局有序、可重放。

- **State**（目标、任务、审批、授权等结构化事实）是物化投影，**永远可由事件流确定性重建**（rebuild）。
- **Memory**（从对话与事实推导的信念）**必须可重新推导**（re-derive）：依据事件流与来源事实再次得出语义一致的记忆；
  换用不同抽取模型时，不要求与旧投影字节级相同，但须保持对用户事实的忠实，且推导过程可审计、可重复触发。

这条原则不是「有数据库」，而是「真相只有一份，且只追加」。

### P2 · AI Agent 不可直达真相源与外部世界

**治理域内的 AI Agent** 不得直接访问真相源（事件日志及其治理投影）与外部世界
（文件系统 / 网络 / 邮件 / Shell 等）。Agent 对世界的改变必须经 Runtime 的受管通道。

**App 不是 Agent。** 仪表盘、设置页、导入向导等产品界面可按需要访问存储与展示数据；
它们不得冒充真相层，也不得替 Agent 绕过治理边界。这条区分避免 endless debate：
「Dashboard 能不能查数据库？」——可以；「Brain 能不能裸写 goals 表？」——不可以。

### P3 · Agent 副作用必须可审计、可追溯

任何 Agent 产生的副作用（改变治理域状态、调用能力、影响外部世界）必须在治理边界内完成，
并在事件流中留下可追溯的因果链。不存在隐式、无审计的 Agent 副作用通道。

### P4 · Capability 必须授权，且 Fail-Closed

能力调用必须经过统一的授权裁决。授权决策基于**类型化的身份**而非裸字符串或隐式信任。
当无法判定一个 Agent 是否被授权时，默认**拒绝**（fail-closed），而非放行（fail-open）。
授权关系本身应可重建、可审计、可撤销。

### P5 · Agent 可替换，接入语义应稳定

Runtime 必须提供**稳定的 Agent 接入契约**：换底层模型、换 Agent 实现、换外部 Agent 框架，
不应迫使用户放弃数据主权或治理承诺。具体接口形态可以演进；**「Agent 可被替换而不破坏
用户数据与治理语义」** 这一承诺不可放弃。North Star 不绑定某一版「ABI」命名或版本号。

### P6 · 治理优先于功能（Governance over Features）

Runtime 存在的目的是**治理个人代理的行为**，不是堆叠功能。功能是 Apps，治理是 Runtime。
当功能与治理边界冲突时，治理胜出。Runtime 提供事实与事件读取；解释、叙事、评分、
仪表盘属于 Apps 层，不属于治理核心。

### P7 · 数据主权不可侵犯

用户对自己的人生数据拥有完全主权：导出、迁移、重建、销毁的权力始终可执行，
且必须由 Runtime 无条件支持。任何让用户数据被平台锁定的设计都被拒绝。

### P8 · 事件溯源约束真相层，而非全系统

事件溯源（Event Sourcing）是**真相层（Truth Layer）的约束**，不是「全库每一张表都必须
事件溯源」。治理域内的人生事实、状态投影、记忆与授权关系，以 Event Log 为权威来源；
应用层缓存、配置、任务队列等辅助状态允许直接读写，只要它们不冒充真相、不被 Agent
误当作可绕过的写入捷径。这一原则划清了「什么是 Runtime 必须保证的真相」与「什么是
产品可以接受的工程便利」之间的界线。

## 4. Non-Goals（明确不做的事）

声明「不做什么」与声明「做什么」同等重要。以下事项**不是**本项目的目标，
即便它们在一般 AI 项目中很常见。它们不应被偷偷塞回 ROADMAP。

### NG1 · 不做分布式 / 微服务

本项目是**单用户、单进程、本地优先**的运行时。治理边界是逻辑边界，不是网络边界。
「分布式一致性」「多节点共识」「服务网格」不在愿景范围内。单机完全可以是一个合法的 Runtime。

### NG2 · 不做通用 PaaS / Agent 编排平台

本项目不为「任意第三方 Agent 应用」提供托管平台。垂直场景（个人记忆、目标、收件箱、
审批治理）验证原语，再按需泛化；但「先造通用平台再找用户」的方向被明确拒绝。

### NG3 · 不追求接入所有 Agent 框架

接入契约的**治理语义**稳定即可。是否能跑 LangChain / AutoGen / CrewAI 不是核心目标。
新 Agent 通过身份与能力声明接入，而非改造治理核心。

### NG4 · 不做多用户隔离

单用户是设计前提。多租户、用户间隔离、跨用户权限不在愿景内。
「为家庭/团队共享」是未来可能的演化，但不是当前 North Star。

### NG5 · 不做网络级沙箱 / 完整 PII 脱敏

安全模型建立在**本地单用户**前提上：API 认证是 opt-in 的 Bearer token；
SSRF 防御、Shell 白名单、文件系统边界是工具层缓解，不是网络沙箱。
LLM 出站仅做审计，不声称完整 PII 脱敏——这是有意的范围界定，不是缺陷。

### NG6 · 不追求全系统事件溯源

见 **P8**。North Star 拒绝的是「要么全盘 Event Sourcing，要么不算 Runtime」的虚假二元。
我们追求的是：**真相层**事件溯源，而非每一张辅助表都进事件流。

### NG7 · 不为短期可用性牺牲边界

当快速交付某个功能需要破坏 P1–P8 任一原则时，选择**不做该功能**，或将其降级为
明确的、有 Issue 追踪的、可收敛的偏差。临时绕过可以存在，但必须可见、可审计、可消除，
绝不能成为隐性的永久架构。

### NG8 · 不追求 AI 自主性（Autonomy）

Runtime 的目标是**治理 Agent**，不是最大化 Agent 自主性。Agent 是否足够聪明属于模型问题；
Runtime 只负责边界、审计、治理与主权——不负责自我进化、自主规划或完全自动化决策。

许多 AI 项目后期会为「更 Autonomous」而开始绕过治理。本项目的立场相反：

```text
Governance > Autonomy
```

自主性可以在 Apps 层实验；治理边界不可为自主性让路。

## 5. 如何使用本文档

- **写新功能时**：先问「这与哪条 Core Principle 相关？是否违反任何 Non-Goal？」
- **写 ROADMAP 时**：每个阶段必须能回溯到本文件的 Mission 或某条 Principle。
- **评审架构时**：Core Principles 是合并 / 拒绝的硬标准，不接受「特殊情况通融」。
- **发生争议时**：本文档优先于任何 ADR、SPEC、README 或个人偏好。
- **可验证的「永远不能破坏」**：见 [INVARIANTS.md](INVARIANTS.md)——North Star 说目标，Invariants 说工程约束。

本文档刻意不提及 SQLite、ChromaDB、FastAPI、具体模型名、具体 Agent 实现等
任何实现细节。这些是实现选择，记录在 [CURRENT_STATE.md](CURRENT_STATE.md)，
不属于 North Star。

**v1.0 之后**：若需调整愿景，走显式「宪法修订」流程（新版本号 + 变更理由），
不在本文件上持续堆叠新原则。
